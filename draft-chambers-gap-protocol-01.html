Abstract

The Generic Authorization Protocol (GAP) defines a framework for cryptographically binding, verifying, and enforcing authorization decisions for autonomous software agents. GAP addresses the problem of AI agents operating with unbounded or opaque permissions by requiring each agent action to carry a signed authorization envelope that includes the agent's identity, the target resource, the requested action, parameters, and a verifiable assurance level. The protocol supports both JSON Web Signature (JWS) and CBOR Object Signing and Encryption (COSE) signing formats, enabling deployment across both web-API and constrained-device environments. Version 1.4.1 introduces normative corrections for payload canonicalization, nonce entropy requirements, replay-window semantics, action-parameter constraints, assurance-level validation, algorithm agility, and updated signing-scope definitions. GAP is designed to interoperate with existing identity frameworks (OAuth 2.0, SPIFFE, X.509) without requiring replacement of deployed infrastructure.

1. Introduction

The rapid proliferation of autonomous software agents across distributed computing environments has introduced a fundamental challenge to authorization architectures. AI-powered agents now routinely execute multi-step workflows that span organizational boundaries, invoke external APIs, modify persistent state, and orchestrate actions across heterogeneous services. These agents operate on behalf of human users, organizational principals, or other agents in complex delegation chains that traditional access-control models were not designed to accommodate.

The current state of agent authorization suffers from what this specification terms the "authorization gap." Agents typically inherit ambient credentials — API keys, OAuth 2.0 bearer tokens, service-account credentials — that grant broad, coarse-grained access without per-action accountability. When an agent presents a bearer token to an API endpoint, the endpoint can verify that the token is valid and that it carries certain scopes, but the endpoint cannot determine which specific action the agent intends to perform, whether that action was explicitly authorized by the principal, or what level of identity assurance the agent possesses. This opacity creates risk: a compromised or misconfigured agent can take unauthorized actions that are indistinguishable from legitimate ones at the transport layer.

The Generic Authorization Protocol (GAP) addresses this gap by establishing a simple, foundational principle: every agent action MUST carry a self-contained, cryptographically signed authorization envelope. The envelope binds the agent's identity, the principal's identity, the target resource, the specific action, the action parameters, and a verifiable assurance level into a single signed data structure. Enforcement Points intercept these envelopes, verify the cryptographic signatures, evaluate policy, and produce allow, deny, or hold decisions before the action reaches the Resource Server.

GAP employs a two-layer architecture. Layer 1 (Identity Binding) ensures that each agent possesses a verifiable identity tied to a cryptographic key pair. The identity may be provisioned through any supported identity framework — OAuth 2.0 [RFC6749], SPIFFE, X.509, or custom schemes — and GAP does not mandate a specific identity provider. Layer 2 (Policy Enforcement) evaluates the signed envelope against local or centralized policy. The policy engine is implementation-specific; GAP defines the envelope format, signing mechanics, and verification procedures that the policy engine consumes.

This specification defines the Authorization Envelope structure, canonicalization procedures, signing and verification rules for both JSON Web Signature (JWS) [RFC7515] and CBOR Object Signing and Encryption (COSE) [RFC9052] formats, nonce and replay-protection semantics, assurance-level validation, algorithm-agility framework, and transport binding for HTTP. GAP is designed to layer atop existing identity and transport infrastructure without requiring replacement of deployed systems.

This document does not cover: key distribution or provisioning, identity-provider protocols, transport-layer security (TLS configuration), policy language or evaluation semantics, or inter-domain trust federation. These topics are deferred to companion specifications or deployment profiles.

1.1. Changes from v1.4.0 to v1.4.1

Version 1.4.1 introduces the following normative corrections and clarifications:

  • Added mandatory canonicalization via JSON Canonicalization Scheme (JCS, RFC 8785) and CBOR Deterministic Encoding (RFC 8949) — see Section 4.
  • Clarified JWS detached-payload mode with b64:false semantics per RFC 7797 and COSE_Sign1 payload handling rules — see Section 5.2 and Section 5.3.
  • Introduced action_params structural constraints: maximum nesting depth of 8, maximum serialized size of 64 KiB, type restrictions, circular-reference prohibition, and key-uniqueness requirement — see Section 6.2.
  • Raised nonce entropy floor to 128 bits from a Cryptographically Secure Pseudo-Random Number Generator (CSPRNG) — see Section 8.1.
  • Added normative assurance_level validation with enumerated values and downgrade-resistance rules — see Section 7.
  • Defined replay-window semantics with a RECOMMENDED 300-second default and minimum/maximum bounds — see Section 8.2.
  • Introduced algorithm-agility framework with mandatory-to-implement cipher suites, a deprecation lifecycle, and post-quantum considerations — see Section 9.
  • Expanded signing-scope definition to include all 13 REQUIRED envelope fields plus 4 OPTIONAL fields, with explicit prohibition of partial inclusion — see Section 5.4.

2. Terminology

2.1. Key Words

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in BCP 14 [RFC2119] [RFC8174] when, and only when, they appear in all capitals, as shown here.

2.2. Definitions

Agent: An autonomous software entity that performs actions on behalf of a principal (human user, organization, or another agent). An Agent possesses a cryptographic key pair and is identified by a unique URI.

Principal: The human or organizational entity that delegates authority to an Agent. The Principal is identified by a URI and is the ultimate source of authorization for the Agent's actions.

Authorization Envelope: The complete signed data structure that accompanies each agent action, containing identity claims, action descriptors, temporal bounds, a nonce, assurance-level indicators, and cryptographic proof. The envelope is the atomic unit of authorization in GAP.

Action Descriptor: The tuple (target, action, action_params) identifying the specific operation requested. Together these three fields fully describe what the Agent intends to do and where.

Signing Scope: The ordered set of envelope fields included in the cryptographic signature computation. The signing scope determines which fields are integrity-protected and which are not.

Assurance Level: A categorical indicator of the strength of identity verification applied to the Agent. Enumerated values: "self-asserted", "domain-validated", "organization-validated", "hardware-bound". See Section 7 for definitions.

Enforcement Point (EP): A component that intercepts agent actions, verifies authorization envelopes, evaluates policy, and produces allow/deny/hold decisions. The EP sits between the Agent and the Resource Server in the request path.

Replay Window: The time interval during which a nonce MUST be retained by an Enforcement Point to detect duplicate submissions. See Section 8.2.

Nonce: A unique, unpredictable value included in each Authorization Envelope to prevent replay attacks. The nonce MUST contain at least 128 bits of entropy generated by a CSPRNG.

Canonicalization: The deterministic transformation of a data structure into a byte sequence suitable for signing. GAP mandates JCS (RFC 8785) for JSON and CBOR Deterministic Encoding (RFC 8949) for CBOR. See Section 4.

3. Protocol Overview

3.1. Architecture

The GAP architecture involves three principals:

  • Agent: The autonomous software entity initiating actions. The Agent constructs and signs Authorization Envelopes.
  • Enforcement Point (EP): The intermediary that intercepts requests, verifies envelopes, evaluates policy, and issues authorization decisions.
  • Resource Server: The system hosting the target resource or API that processes the authorized action.

The protocol flow proceeds as follows:

  +-------+                  +--------------------+                  +-----------------+
  | Agent |---[Signed Env]-->| Enforcement Point  |---[Verified]--->| Resource Server |
  +-------+                  | (Verify + Policy)  |                  +-----------------+
                             +--------------------+
                                     |
                              Allow / Deny / Hold

The Agent constructs an Authorization Envelope containing all required fields, canonicalizes and signs it, then attaches the signed envelope to the outgoing request. The Enforcement Point intercepts the request, extracts the envelope, verifies the signature, validates structural and temporal constraints, evaluates policy, and produces an allow, deny, or hold decision. On allow, the EP forwards the request to the Resource Server with verified envelope metadata. On deny, the EP returns an error response to the Agent. On hold, the EP may defer the decision pending additional input (e.g., human approval).

3.2. Authorization Envelope Structure

The Authorization Envelope is a JSON object (or CBOR map) with the following REQUIRED fields:

{
  "gap_version":       "1.4.1",
  "envelope_id":       "<UUID v4>",
  "agent_id":          "<URI>",
  "principal_id":      "<URI>",
  "issued_at":         "<RFC 3339 timestamp>",
  "expires_at":        "<RFC 3339 timestamp>",
  "nonce":             "<base64url-encoded, min 128 bits>",
  "assurance_level":   "<enumerated string>",
  "target":            "<URI of resource server or endpoint>",
  "action":            "<string action identifier>",
  "action_params":     { ... },
  "signing_algorithm": "<algorithm identifier>",
  "key_id":            "<key identifier (kid)>"
}

Field Definitions:

  • gap_version (string, REQUIRED): The GAP protocol version. For this specification, the value MUST be "1.4.1". Enforcement Points MUST reject envelopes with unrecognized version strings.
  • envelope_id (string, REQUIRED): A UUID v4 uniquely identifying this envelope. Implementations MUST generate a fresh UUID for each envelope. The envelope_id is used for logging, auditing, and correlation.
  • agent_id (string, REQUIRED): A URI identifying the Agent. Examples include URN-based identifiers (urn:gap:agent:example:bot-1), SPIFFE IDs (spiffe://trust-domain/agent/bot-1), or HTTPS-based identifiers.
  • principal_id (string, REQUIRED): A URI identifying the Principal on whose behalf the Agent acts. This binds the agent action to a delegating authority.
  • issued_at (string, REQUIRED): An RFC 3339 timestamp indicating when the envelope was created. MUST be expressed in UTC (with "Z" suffix). Used for replay-window computation (Section 8.2).
  • expires_at (string, REQUIRED): An RFC 3339 timestamp indicating when the envelope expires. MUST be after issued_at. Enforcement Points MUST reject envelopes where the current time exceeds expires_at.
  • nonce (string, REQUIRED): A base64url-encoded value with at least 128 bits of entropy from a CSPRNG. See Section 8.1 for full requirements.
  • assurance_level (string, REQUIRED): One of the four enumerated assurance-level values defined in Section 7.1. Any other value MUST cause rejection.
  • target (string, REQUIRED): A URI identifying the resource server or specific endpoint the Agent intends to access.
  • action (string, REQUIRED): A string identifying the specific operation the Agent requests (e.g., "query_weather", "create_order", "delete_record"). Action identifiers SHOULD be drawn from a documented action vocabulary for the target service.
  • action_params (object, REQUIRED): A JSON object containing parameters for the requested action. MAY be an empty object {} if the action requires no parameters. See Section 6 for structural constraints.
  • signing_algorithm (string, REQUIRED): The identifier of the algorithm used to sign the envelope. For JWS, this is the JWA algorithm name (e.g., "ES256"). For COSE, this is the COSE algorithm identifier. See Section 9 for the algorithm registry.
  • key_id (string, REQUIRED): The identifier of the signing key (kid). Used by the Enforcement Point to locate the corresponding public key for verification.

The following OPTIONAL fields MAY be included:

  • delegation_chain (array): An ordered array of delegation tokens establishing a chain of delegated authority from the Principal to the Agent. Each element represents one delegation step.
  • scope_constraints (object): Additional scope restrictions imposed by the Principal or an intermediate delegator, narrowing the Agent's permitted actions.
  • binding_metadata (object): Transport or channel binding data (e.g., TLS channel binding tokens) linking the envelope to a specific transport session.
  • extensions (object): Registered extension fields for protocol extensions. Keys MUST be drawn from a future GAP Extensions Registry.

3.3. Protocol Flow

The complete protocol flow proceeds through the following steps:

  1. Credential Acquisition: The Agent obtains its cryptographic key pair and any necessary identity attestations from an identity provider. This step is out of scope for GAP.
  2. Envelope Construction: The Agent constructs the Authorization Envelope with all REQUIRED fields populated. The nonce is generated fresh per Section 8.1. The issued_at and expires_at are set to appropriate values.
  3. Canonicalization: The Agent canonicalizes the envelope per Section 4. For JSON, this means applying JCS (RFC 8785). For CBOR, this means applying CBOR Deterministic Encoding (RFC 8949).
  4. Signing: The Agent signs the canonical byte sequence per Section 5, producing a JWS Compact Serialization (with detached payload) or a COSE_Sign1 structure.
  5. Transmission: The Agent transmits the request to the Enforcement Point. The signed token is placed in the GAP-Authorization HTTP header (Section 10.1). The envelope body is transmitted as the request body.
  6. Verification: The Enforcement Point extracts the envelope and performs the following checks in order:
    1. Structural validity: All REQUIRED fields are present and have correct types.
    2. Action-params validation: action_params conforms to the structural constraints of Section 6.2.
    3. Temporal validity: issued_at <= now <= expires_at, within clock-skew tolerance (Section 8.3).
    4. Nonce uniqueness: The (agent_id, nonce) tuple is not present in the replay cache (Section 8.2).
    5. Signature verification: The cryptographic signature is valid per Section 5.
    6. Assurance-level validation: The assurance_level meets the minimum required for the target/action combination (Section 7).
    7. Policy evaluation: Implementation-specific policy rules are applied.
  7. Decision: The EP issues one of three decisions: allow (forward the request), deny (return an error), or hold (defer pending additional input such as human approval).
  8. Forwarding: On allow, the EP forwards the request to the Resource Server, optionally attaching verified envelope metadata (e.g., agent_id, assurance_level, policy decision identifier) for the Resource Server's use.

4. Canonicalization

4.1. Requirement

Before signing or verifying an Authorization Envelope, implementations MUST serialize the envelope into a canonical byte sequence. The canonical form is the sole input to the signature computation. Signing non-canonical representations is a protocol violation and MUST cause verification failure at the Enforcement Point.

Canonicalization ensures that two implementations processing the same logical envelope produce identical byte sequences, which is essential for signature interoperability. Without canonicalization, semantically equivalent envelopes (e.g., differing only in whitespace or key order) would produce different signatures, breaking cross-implementation verification.

4.2. JSON Canonicalization Scheme (JCS)

Implementations using JSON MUST apply the JSON Canonicalization Scheme as defined in RFC 8785 [RFC8785]. JCS produces deterministic JSON output with the following properties:

  • Sorted object member keys: All object members are sorted by key in lexicographic Unicode code-point order. Sorting is applied recursively to nested objects.
  • No insignificant whitespace: No whitespace is emitted between tokens (no spaces after colons or commas, no indentation, no newlines).
  • Deterministic number serialization: Numbers are serialized per ECMAScript's Number.toString() specification — no trailing zeros, no positive-sign prefix, no unnecessary decimal points for integers, and exponential notation only when required by the specification.
  • UTF-8 encoding without BOM: The output is encoded as UTF-8. A Byte Order Mark (BOM) MUST NOT be prepended.
  • String escaping per RFC 8785: Strings use the minimal escaping defined by JCS. Characters U+0000 through U+001F are escaped using \uXXXX notation. Backslash and double-quote are escaped. All other characters are represented literally in UTF-8.

The canonical byte sequence is the UTF-8 encoding of the JCS-serialized envelope. This byte sequence is used directly as the payload input for JWS signing (Section 5.2).

4.3. CBOR Canonicalization

For COSE-based deployments, implementations MUST serialize the Authorization Envelope using CBOR Deterministic Encoding as specified in Section 4.2 of RFC 8949 [RFC8949]. The following rules apply:

  • Map keys MUST be sorted in bytewise lexicographic order of their deterministic CBOR encoding.
  • Indefinite-length encodings MUST NOT be used. All data items MUST use definite-length encoding.
  • Integers MUST be encoded in the smallest number of bytes that can represent the value.
  • Floating-point values SHOULD be avoided where integer representation is sufficient. When used, preferred serialization per Section 4.2.1 of RFC 8949 applies.

The canonical CBOR byte sequence is used directly as the payload input for COSE signing (Section 5.3).

4.4. Canonicalization Scope

The canonicalization MUST be applied to the complete Authorization Envelope object, including all REQUIRED and OPTIONAL fields present at the time of signing. Fields added after signing (e.g., transport headers, EP annotations, routing metadata) MUST NOT be included in the canonical form.

Implementations MUST NOT selectively exclude envelope fields from canonicalization. The canonical form is a complete, faithful representation of the envelope as constructed by the Agent at signing time.

5. Signing and Verification

5.1. General Requirements

  • Every Authorization Envelope MUST be signed by the Agent using the private key corresponding to the key_id identified in the envelope.
  • The signature MUST cover the canonical byte sequence produced per Section 4. Signing any other representation of the envelope is a protocol violation.
  • Two signing formats are defined: JWS (Section 5.2) for JSON/HTTP environments and COSE (Section 5.3) for constrained or CBOR-native environments.
  • Implementations MUST support at least one signing format. Implementations that support both MUST correctly negotiate the format via the gap_format header or out-of-band configuration.
  • The signing algorithm used MUST be one of the algorithms listed in the GAP Algorithm Registry (Section 9) with a lifecycle state of MANDATORY, RECOMMENDED, or OPTIONAL. Algorithms in DEPRECATED state MUST NOT be used for new signatures. Algorithms in PROHIBITED state MUST NOT be used or accepted.

5.2. JWS Signing (RFC 7515)

5.2.1. Detached Payload Mode

When using JWS, implementations MUST use the JWS Compact Serialization with detached payload as specified in Appendix F of RFC 7515 [RFC7515]. The payload (the canonical envelope byte sequence) MUST NOT appear in the JWS token itself; instead, the canonical bytes are provided out-of-band — in the HTTP request body or a designated header.

Rationale: Detached payloads avoid double-encoding the envelope (once in the request body and again in the JWS payload segment). They also ensure that the Enforcement Point can independently reconstruct the canonical form from the envelope fields it receives, compare it to the signed content, and detect tampering of any field.

5.2.2. JWS Header Requirements

The JWS Protected Header MUST contain the following parameters:

ParameterValueRequirement
algThe signing algorithm (Section 9)REQUIRED
kidThe key identifier matching the envelope's key_idREQUIRED
b64falseREQUIRED
crit["b64"]REQUIRED
gap_ver"1.4.1"REQUIRED

The b64 header parameter MUST be set to false, and "b64" MUST appear in the crit array, per RFC 7797 [RFC7797] Section 5. This ensures the payload bytes are used directly in the signing input without base64url encoding, enabling the EP to verify the signature against the raw canonical bytes.

5.2.3. JWS Signing Input

The JWS Signing Input is computed as:

ASCII(BASE64URL(UTF8(JWS Protected Header))) || '.' || JCS_Canonical_Bytes

Where JCS_Canonical_Bytes is the raw canonical byte sequence produced by Section 4.2, not base64url-encoded, per the b64:false semantics of RFC 7797. The dot ('.') separator is the ASCII period character (0x2E).

5.2.4. JWS Verification

Enforcement Points MUST perform the following steps to verify a JWS-signed envelope:

  1. Extract the JWS Protected Header from the detached JWS token (the first segment before the first dot).
  2. Base64url-decode and parse the Protected Header as JSON.
  3. Verify that gap_ver matches the expected protocol version ("1.4.1").
  4. Verify that b64 is false and that "b64" appears in the crit array.
  5. Reconstruct the canonical byte sequence from the received envelope per Section 4.2 (JCS).
  6. Compute the JWS Signing Input per Section 5.2.3 using the reconstructed canonical bytes.
  7. Retrieve the public key identified by the kid parameter.
  8. Verify the signature (the third segment of the JWS token) against the Signing Input using the public key and the algorithm specified in alg.
  9. If any step fails, reject the envelope with the appropriate error code (Section 10.2).

5.3. COSE Signing (RFC 9052)

5.3.1. COSE_Sign1 Structure

When using COSE, implementations MUST use the COSE_Sign1 structure (single signer) as defined in Section 4.2 of RFC 9052 [RFC9052]. The COSE_Sign1 CBOR array contains four elements:

COSE_Sign1 = [
  protected   : bstr,    ; CBOR-encoded protected header map
  unprotected : map,     ; Empty map {}
  payload     : bstr,    ; Canonical CBOR bytes (Section 4.3)
  signature   : bstr     ; Computed signature
]

The payload MUST be included directly in the COSE_Sign1 structure (i.e., not detached) unless the deployment profile explicitly specifies detached payloads. In detached mode, the payload field is set to nil and the canonical bytes are conveyed separately. Deployment profiles that specify detached COSE payloads MUST document the mechanism for conveying the payload bytes.

5.3.2. COSE Header Parameters

The protected header map MUST contain the following parameters:

LabelParameterValueRequirement
1AlgorithmCOSE algorithm identifier (e.g., -7 for ES256)REQUIRED
4Key IDKey identifier bytesREQUIRED
-65537GAP VersionUTF-8 string "1.4.1"REQUIRED

The GAP Version parameter uses private-use label -65537. Registration in the COSE Header Parameters registry is requested in Section 12.3. The unprotected header map MUST be empty.

5.3.3. Sig_structure for COSE_Sign1

The Sig_structure is computed per Section 4.4 of RFC 9052:

Sig_structure = [
  "Signature1",      ; context string
  protected,         ; CBOR-encoded protected header
  external_aad,      ; External Additional Authenticated Data
  payload            ; Canonical CBOR bytes
]

The external_aad is empty (h'') unless a deployment profile specifies additional authenticated data. The payload is the canonical CBOR byte sequence from Section 4.3. The Sig_structure is CBOR-encoded, and the resulting byte string is the input to the signature algorithm.

5.4. Signing Scope

The signing scope is the set of fields whose values are included in the canonical byte sequence that is signed. The signing scope MUST include ALL of the following fields when present in the envelope:

REQUIRED in signing scope (MUST always be present):

  1. gap_version
  2. envelope_id
  3. agent_id
  4. principal_id
  5. issued_at
  6. expires_at
  7. nonce
  8. assurance_level
  9. target
  10. action
  11. action_params (full canonical form, even if empty object)
  12. signing_algorithm
  13. key_id

OPTIONAL in signing scope (included if present in the envelope):

  1. delegation_chain (array of delegation tokens, if delegated authority)
  2. scope_constraints (additional scope restrictions)
  3. binding_metadata (transport or channel binding data)
  4. extensions (registered extension fields)

Fields MUST be included in the canonical form in their entirety. Partial inclusion — for example, signing action but omitting action_params — is a protocol violation and MUST cause verification failure.

Enforcement Points MUST reject envelopes where the signing scope does not cover all REQUIRED fields. The EP MUST independently reconstruct the canonical form from the received envelope and verify it matches the signed content. If the EP detects that any REQUIRED field was modified or omitted after signing, the envelope MUST be rejected with error gap_canonicalization_mismatch (Section 10.2).

6. Action Parameters

6.1. Structure

The action_params field is a JSON object (or CBOR map) containing parameters specific to the requested action. It is included in the signing scope (Section 5.4) and therefore MUST conform to the canonicalization requirements of Section 4.

The semantics of action_params are defined by the target service. GAP does not prescribe the contents of action_params beyond the structural constraints defined in this section. Target services SHOULD document their expected action_params schemas.

6.2. Structural Constraints

Implementations MUST enforce the following constraints on action_params:

Maximum Nesting Depth: action_params MUST NOT exceed a nesting depth of 8 levels. Depth is counted from the action_params root object (depth 0) inward. Objects or arrays at depth 8 MUST NOT contain further objects or arrays. Scalar values (strings, numbers, booleans, null) at any depth are permitted.

Maximum Serialized Size: The JCS-canonical (or CBOR deterministic) serialization of action_params MUST NOT exceed 65,536 bytes (64 KiB). Enforcement Points MUST reject envelopes where action_params exceeds this limit. This constraint prevents resource exhaustion during parsing and canonicalization.

Allowed Value Types: Values within action_params MUST be one of: string, number (integer or IEEE 754 finite floating-point), boolean, null, array, or object. Special IEEE 754 values (NaN, Infinity, -Infinity) MUST NOT appear. For CBOR deployments: tags MUST NOT be used within action_params unless defined by a registered extension.

No Circular References: action_params MUST NOT contain circular references. Implementations MUST detect and reject circular structures during canonicalization. In practice, JSON inherently disallows circular references, but CBOR implementations using shared-value tags MUST guard against constructed cycles.

Key Uniqueness: All object member keys within action_params (at any depth) MUST be unique strings. Duplicate keys MUST cause rejection. This requirement aligns with RFC 8259 Section 4 ("The names within an object SHOULD be unique") and makes it normative for GAP.

6.3. Validation Ordering

Enforcement Points MUST validate action_params structural constraints BEFORE signature verification. This ordering prevents resource exhaustion from processing cryptographic operations on malformed or oversized payloads. A malicious actor could submit envelopes with pathologically nested or oversized action_params designed to consume verification resources; early structural rejection mitigates this vector.

7. Assurance Levels

7.1. Enumerated Values

GAP defines four assurance levels, ordered from lowest to highest:

LevelValue StringMeaning
1 "self-asserted" Agent identity is claimed but not independently verified. Key pair is self-generated. Suitable for low-risk, sandboxed operations where the consequences of impersonation are minimal.
2 "domain-validated" Agent identity is bound to a verified domain via DNS challenge, ACME challenge, or equivalent mechanism. Key pair is certified by a domain-level Certificate Authority or SPIFFE trust domain.
3 "organization-validated" Agent identity is bound to a verified organizational entity. Key pair is issued by an organization-level Certificate Authority with legal-entity validation (e.g., OV or EV certificate procedures).
4 "hardware-bound" Agent signing key is generated and stored in a hardware security module (HSM), Trusted Platform Module (TPM), or secure enclave. Key extraction is prevented by hardware. The highest assurance level.

7.2. Validation Requirements

  • The assurance_level field MUST contain exactly one of the four enumerated value strings. Any other value MUST cause envelope rejection with error gap_malformed.
  • Enforcement Points MUST validate that the claimed assurance_level is consistent with the presented key material and identity attestation. For example: an envelope claiming "hardware-bound" MUST present an attestation certificate chain rooting in a recognized hardware manufacturer's Certificate Authority. An envelope claiming "domain-validated" MUST present a certificate or SPIFFE SVID bound to a verified domain.
  • Implementations MUST NOT treat assurance_level as a free-form string. Case-sensitive exact matching is REQUIRED. The value "Hardware-Bound" or "HARDWARE-BOUND" MUST be rejected.

7.3. Downgrade Resistance

Once an Enforcement Point has observed an agent operating at a given assurance level, it SHOULD record that level as the agent's "high-water mark." Subsequent envelopes from the same agent_id presenting a lower assurance level SHOULD trigger a policy alert and MAY be rejected, depending on local policy.

This mechanism provides downgrade resistance: an attacker who compromises a software-based signing key cannot silently downgrade from "hardware-bound" to "self-asserted" without detection. The EP's assurance-level history for each agent provides a signal that something has changed and warrants investigation.

Enforcement Points SHOULD log all observed assurance-level transitions for auditing purposes.

7.4. Minimum Assurance Policies

Enforcement Points MAY define minimum assurance-level requirements per target, per action, or per action-target combination. An envelope whose assurance_level does not meet the minimum MUST be rejected with error gap_assurance_insufficient and SHOULD include an indication of the required level in the error response body.

Example: An EP protecting a financial-transactions API might require "hardware-bound" for action: "transfer_funds" but accept "domain-validated" for action: "query_balance".

8. Nonce and Replay Protection

8.1. Nonce Generation

  • The nonce field MUST contain a value with at least 128 bits of entropy, generated by a Cryptographically Secure Pseudo-Random Number Generator (CSPRNG).
  • The nonce MUST be encoded as a base64url string per RFC 4648 [RFC4648] Section 5, without padding characters.
  • A 128-bit (16-byte) random value produces a 22-character base64url string. Implementations MAY use longer nonces (e.g., 256 bits / 32 bytes / 43 characters).
  • Nonces MUST be unique per envelope. Reuse of a nonce with different envelope contents is a protocol violation.
  • Implementations MUST NOT use predictable values (timestamps, counters, sequential identifiers, UUIDs) as the sole source of nonce material. Predictable nonces enable precomputation attacks where an adversary generates valid-looking envelopes before the legitimate agent acts.
  • Implementations MAY combine random bytes with other data (e.g., a timestamp prefix) provided that at least 128 bits are derived from a CSPRNG.

8.2. Replay Window

  • Enforcement Points MUST maintain a replay-detection cache that stores the tuple (agent_id, nonce) for each verified envelope.
  • The RECOMMENDED replay-window duration is 300 seconds (5 minutes).
  • Implementations MAY configure different window durations, but MUST NOT set the window below 60 seconds (to accommodate reasonable clock skew and network latency) and SHOULD NOT exceed 3,600 seconds (to limit cache growth and memory consumption).
  • An envelope whose issued_at timestamp falls outside the replay window — i.e., issued_at < now - replay_window OR issued_at > now + max_clock_skew — MUST be rejected with error gap_expired.
  • If an (agent_id, nonce) tuple matching an incoming envelope already exists in the replay cache, the envelope MUST be rejected with error gap_replay.
  • Cache entries MUST be retained for the full duration of the replay window and SHOULD be purged promptly after expiry to conserve resources.

8.3. Clock Skew Tolerance

  • Implementations MUST allow for clock skew between the Agent and the Enforcement Point.
  • The RECOMMENDED maximum clock-skew tolerance is 30 seconds.
  • Implementations SHOULD use NTP or similar time-synchronization protocols (e.g., PTP, roughtime) to minimize clock skew between components.
  • Clock-skew tolerance is applied when evaluating temporal validity: an envelope is temporally valid if issued_at - max_clock_skew <= now <= expires_at + max_clock_skew.

8.4. Distributed Replay Detection

In distributed EP deployments (e.g., multiple EP instances behind a load balancer), replay caches MUST be synchronized or shared across all EP instances that may receive envelopes from the same agent. Failure to synchronize replay caches creates a vulnerability where an attacker can replay an envelope to a different EP instance that has not yet recorded the nonce.

Common approaches to distributed replay detection include:

  • A shared, replicated cache (e.g., Redis Cluster, Memcached with replication).
  • Consistent hashing to route all envelopes from a given agent_id to the same EP instance (note: this introduces a single-point-of-failure risk).
  • A centralized nonce-check service with low-latency access from all EP instances.

Implementers SHOULD evaluate the latency and consistency characteristics of their chosen approach against their throughput requirements and threat model.

9. Algorithm Agility

9.1. Design Principles

GAP's algorithm agility framework ensures that the protocol can adapt to evolving cryptographic requirements without breaking backward compatibility. The framework provides:

  • A registry of approved algorithms with lifecycle states governing their use.
  • Mandatory-to-implement cipher suites ensuring baseline interoperability across all implementations.
  • A deprecation mechanism for transitioning away from weakened algorithms in a controlled manner.
  • Negotiation rules for environments where Agents and Enforcement Points support multiple algorithms.

9.2. Algorithm Registry

The GAP Algorithm Registry (Section 12.5) contains one entry per approved algorithm. Each entry has the following fields:

FieldDescription
Algorithm IdentifierJWA name (for JWS) or COSE algorithm ID (for COSE)

Lifecycle states are ordered: MANDATORY > RECOMMENDED > OPTIONAL > DEPRECATED > PROHIBITED. Algorithms transition through these states over time as cryptographic best practices evolve.

The following algorithms are MANDATORY for all GAP implementations. Every conforming implementation MUST support both signing and verification for these algorithms.

RECOMMENDED algorithms (implementations SHOULD support):

  • ES384 — ECDSA P-384 with SHA-384 (EC key, minimum 384 bits)
  • PS256 — RSASSA-PSS with SHA-256 (RSA key, minimum 2,048 bits)

DEPRECATED algorithms (MUST NOT be used for new envelopes; MAY be accepted for verification during transition):

  • RS256 — RSASSA-PKCS1-v1_5 with SHA-256. Deprecated due to known padding-oracle concerns and the availability of superior alternatives (PS256, ES256).

PROHIBITED algorithms (MUST NOT be used, MUST NOT be accepted):

  • none — The unsecured algorithm. Permitting "none" would allow unsigned envelopes, completely undermining the protocol's security model.
  • HS256, HS384, HS512 — Symmetric HMAC algorithms. These are inappropriate for the agent-to-EP trust model because they require the Agent and EP to share a secret key, which violates the asymmetric-key assumption of GAP.

9.4. Algorithm Negotiation

When an Agent and Enforcement Point support multiple algorithms, the following negotiation rules apply:

  1. The Agent selects its preferred algorithm and includes it in the signing_algorithm field of the envelope and in the JWS alg header or COSE Algorithm header parameter.
  2. The EP MUST verify using the algorithm specified in the envelope. The EP MUST NOT substitute a different algorithm. Algorithm substitution is a known attack vector (Section 11.6).
  3. If the EP does not support the specified algorithm, it MUST reject the envelope with error gap_algorithm_rejected and SHOULD include a list of its supported algorithms in the error response.
  4. The EP SHOULD advertise its supported algorithms via a discovery endpoint (/.well-known/gap-configuration) or equivalent out-of-band mechanism. The discovery document format is defined in Section 12.4.

9.5. Deprecation Lifecycle

Algorithm deprecation follows a three-phase lifecycle:

  1. Announcement (minimum 6 months): The algorithm is marked DEPRECATED in the GAP Algorithm Registry. Implementations SHOULD begin migrating to alternative algorithms. New deployments SHOULD NOT use the deprecated algorithm for signing.
  2. Signing Prohibition (minimum 12 months after announcement): The algorithm MUST NOT be used for creating new envelopes. Enforcement Points MAY still accept it for verification of previously signed envelopes during the transition period.
  3. Full Prohibition: The algorithm is marked PROHIBITED in the registry. Enforcement Points MUST reject envelopes using this algorithm. No grace period applies.

The minimum timelines above allow the ecosystem sufficient time to transition. Specific deprecation timelines for each algorithm are recorded in the GAP Algorithm Registry.

9.6. Post-Quantum Considerations

This specification does not mandate post-quantum algorithms at this time. The current mandatory-to-implement algorithms (ES256, EdDSA) are based on elliptic-curve cryptography and are believed to be secure against classical adversaries but are vulnerable to quantum attack using Shor's algorithm.

Implementations SHOULD monitor NIST post-quantum standardization efforts, specifically:

  • ML-DSA (Module-Lattice-Based Digital Signature Algorithm, FIPS 204 [FIPS204])
  • SLH-DSA (Stateless Hash-Based Digital Signature Algorithm, FIPS 205 [FIPS205])

Future versions of GAP will incorporate ML-DSA or SLH-DSA as mandatory-to-implement algorithms when ecosystem readiness permits — specifically, when major cryptographic libraries, hardware security modules, and deployment platforms provide stable support for these algorithms.

10. Transport Binding

10.1. HTTP Transport

When GAP is used over HTTP, the following transport binding applies:

  • The signed envelope MUST be transmitted in the GAP-Authorization HTTP header field. The header value is the JWS Compact Serialization (with detached payload, i.e., header..signature) or the base64url-encoded COSE_Sign1 structure.
  • The envelope body (from which the canonical form is reconstructed by the EP) MUST be transmitted as the HTTP request body.
  • For JWS-based deployments, the Content-Type MUST be application/gap+json.
  • For COSE-based deployments, the Content-Type MUST be application/gap+cbor.
  • Implementations SHOULD use HTTP/2 or HTTP/3 to benefit from header compression and multiplexing.
  • Transport-layer encryption (TLS 1.3 or later) SHOULD be used to protect envelopes in transit. Note that TLS provides confidentiality and transport integrity but does not replace GAP's application-layer authorization semantics.

11. Security Considerations

11.1. Threat Model

GAP is designed to mitigate the following threats:

  • Replay Attacks: An adversary captures a valid signed envelope and resubmits it to the EP. Mitigated by nonce uniqueness and the replay-window cache (Section 8).
  • Signature Forgery: An adversary attempts to create a valid envelope without possession of the Agent's private key. Mitigated by mandatory asymmetric digital signatures using approved algorithms (Section 9).
  • Parameter Tampering: An adversary modifies envelope fields (e.g., changing action from "read" to "delete") after signing. Mitigated by the signing scope covering all REQUIRED fields (Section 5.4) and canonical verification at the EP.
  • Assurance-Level Downgrade: An adversary who has compromised a software key attempts to operate at a lower assurance level to avoid detection. Mitigated by downgrade-resistance monitoring (Section 7.3).
  • Algorithm Substitution: An adversary modifies the algorithm identifier to force verification with a weaker algorithm. Mitigated by including the algorithm in the signed scope and by prohibiting algorithm inference (Section 11.6).
  • Timing Attacks: An adversary observes verification duration to infer information about the key or signature. Mitigated by using constant-time signature verification routines, as mandated by the underlying JWS and COSE libraries.
  • Envelope Stripping: An adversary removes the GAP-Authorization header entirely, causing the request to bypass authorization. Mitigated by EP policy: EPs MUST reject requests without valid GAP envelopes for protected resources.

11.2. Canonicalization Security

Deterministic canonicalization is essential to GAP's security model. Without it, two valid JSON serializations of the same logical object (e.g., differing in whitespace or key order) would produce different signature inputs, creating multiple valid signatures for the same logical envelope. An attacker could exploit this ambiguity by re-serializing an envelope in a different form, changing the canonical bytes without changing the logical content, and causing signature verification to fail or produce inconsistent results.

JCS (RFC 8785) and CBOR Deterministic Encoding (RFC 8949) eliminate this class of vulnerability by ensuring that any two compliant implementations produce identical byte sequences for the same logical input. Implementations MUST use well-tested, specification-compliant canonicalization libraries. Custom implementations are strongly discouraged due to the subtlety of Unicode normalization, number serialization, and key-sorting edge cases.

11.3. Nonce Security

The 128-bit entropy requirement for nonces provides strong collision resistance. By the birthday bound, the probability of a nonce collision is less than 2^-64 after 2^32 (approximately 4.3 billion) envelopes. This is well within acceptable bounds for any practical deployment.

The CSPRNG requirement prevents prediction and precomputation attacks. If nonces were predictable (e.g., sequential counters), an adversary could precompute envelopes with future nonces, undermining replay protection. Implementations MUST use the operating system's CSPRNG (e.g., /dev/urandom, CryptGenRandom, getrandom(2)).

11.4. Replay Protection Limitations

Replay windows create a fundamental tradeoff between cache cost and protection duration. A 300-second window requires storing up to 300 × (peak_envelopes_per_second) nonce entries. For a system processing 10,000 envelopes per second, this is approximately 3 million entries — manageable for in-memory caches but significant for resource-constrained environments.

Distributed cache synchronization introduces latency and potential consistency gaps. During the propagation delay, a replay may succeed against a different EP instance. Operators SHOULD size replay caches for expected throughput and SHOULD monitor cache hit rates to detect replay-attack attempts.

11.5. Key Management

Key Compromise: If an Agent's private key is compromised, all envelopes signed with that key are suspect. Enforcement Points SHOULD support key revocation mechanisms (e.g., CRL, OCSP, or out-of-band revocation notification). Upon learning of a compromise, the affected key MUST be revoked and all EP instances MUST be updated to reject the compromised key_id.

Key Rotation: Agents SHOULD rotate signing keys periodically (e.g., every 90 days). During rotation, EPs SHOULD support overlapping key validity periods to allow a smooth transition. The Agent may sign with the new key while the EP still accepts both old and new keys.

Hardware-Bound Keys: Hardware-bound keys (HSM, TPM, secure enclave) mitigate extraction risk by ensuring the private key never leaves the hardware boundary. This is the strongest form of key protection and corresponds to the "hardware-bound" assurance level (Section 7.1).

11.6. Algorithm Substitution Attacks

The signing algorithm is included in the signing scope (via the signing_algorithm field in the envelope and the alg header in JWS/COSE). This prevents algorithm substitution attacks where an adversary modifies the algorithm identifier to force verification with a weaker algorithm.

Enforcement Points MUST NOT infer the algorithm from the key type alone. For example, an EP MUST NOT see an EC P-256 key and assume ES256 — the algorithm MUST be taken from the envelope's signing_algorithm field and the JWS/COSE header, and both MUST match.

11.7. Denial of Service

The action_params size and depth limits (Section 6.2) prevent resource exhaustion during parsing and canonicalization. Without these limits, an attacker could submit envelopes with deeply nested or extremely large parameter objects designed to consume CPU and memory during JSON/CBOR parsing.

Structural validation before cryptographic verification (Section 6.3) reduces the cost of processing malformed envelopes. Cryptographic signature verification is computationally expensive; performing it only after confirming structural validity ensures that malformed payloads are rejected cheaply.

Rate limiting SHOULD be applied per agent_id at the Enforcement Point to prevent a compromised or malicious agent from flooding the EP with verification requests.

11.8. Privacy Considerations

Authorization envelopes contain agent_id and principal_id, which may be sensitive identifiers. If envelopes are transmitted without transport encryption, these identifiers are exposed to network observers.

  • Transport encryption (TLS 1.3 or later) SHOULD be used to protect envelopes in transit.
  • Logs containing envelopes SHOULD redact or hash principal_id per applicable data-protection requirements (e.g., GDPR, CCPA).
  • Implementations SHOULD consider using pseudonymous or pairwise identifiers for agent_id and principal_id where full identification is not required.
  • Replay-detection caches contain (agent_id, nonce) tuples, which could reveal agent activity patterns. Caches SHOULD be protected with appropriate access controls.
